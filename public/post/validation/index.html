Validation

Last updated on 2022-07-01 6 min read

Let’s discuss validation

tl;dr

In fields adjacent to data science and machine learning, the word validation conveys very different concepts and invokes very different expectations. Unless we, the data professionals and machine learning engineers, are careful, this clash of terminology can complicate communication or cause misunderstandings. By understanding the expectations of the field we are applying data science to and by avoiding the term validation, we can improve our collaborations and applications.

Validation is awesome and necessary

When doing validation in a pure machine learning model development context, we are trying to assess how well our model generalises to data that it has not seen before. The metric to estimate is generalisation error and the means to estimate it is cross validation (CV). With CV, we leave out pieces of our data and train the model on the rest, in order to simulate unseen data. It is also common practice to do double CV, where generalisation error averaged across a list of models or model configurations as well. This gives a neat metric for model selection. Any good machine learning developer will tell you that the predictions of even the best model will be no better than the data used for training. In the same way, the generalisation error is only as good as the data it was estimated on. So in the worst case, the output of the validation step for machine learning model development is just an estimate of how well any machine learning model might generalise to the data at hand. That is not to say that CV is a bad or even lacking method. In fact, it is quite the opposite. Any good machine learning model development flow should include one or more validation steps with cross validation. However, the outputs of validation in a pure machine learning model development context are really only interesting for other ML developers and, more importantly, they do not conform to the standard that is suggested by the term validation in many other contexts.

What is Validation in other fields

Consider the following hypothetical situations.

An integer feature for our model suddenly receives a float input. A categorical feature in our model receives a category that it has not seen before.

For me these are not unfamiliar complications, and we would probably expect the model to crash when these complications happen. Returning an error rather than a prediction might even be the desired behaviour in such cases. Consider, however, these situations:

A feature normalised between 0 and 1 in the training data receives an input of -2. The input image is corrupted and half the pixels are randomly 0. In prediction for a single object, 19 out of 20 input features receive extreme values, say above the 99.9th percentile.

In all of these cases, our hypothetical model would happily return predictions. Would those predictions be good? We cannot say for sure, but probably not. Is that the model’s fault? Absolutely not. However, in any real life application the model would probably receive bad credit for providing such poor predictions. Of course we as developers build logic at both ends of the model to handle input sanitation and output presentation, so as to avoid such a situation. I assert that, even in the most simple applications of machine learning, there is a software layer around the model.

It gets worse. Consider these cases:

Someone bumps into the camera that takes pictures for predictions. Now all images are at a new angle. The light bulb providing light for the pictures burns out. Now all images are a bit darker.

Just as there is a software layer around the machine learning model, so too is there often a hardware layer responsible for generating the data in the first place.

It is clear that a machine learning model does not exist in a vacuum. Models are part of a system of components, and the model will never be able to handle all the strange states such a system might find itself in. Consider the case of the defect light bulb. We could develop a model that works somewhat well even in dimmer conditions, but I assert that it is faster and better in the long run to just change the bulb before it breaks.

The downside to that is that now the process of changing a light bulb has become part of the machine learning system. That might not feel like a fitting part of a glamorous AI system. However, consider that in order for our model performance estimates to hold in the final application, we need to make sure that all components work together to deliver data that is within the expected variance.

Orchestrating all the components of a machine learning system is hard, but it is such integrative thinking and testing that is often expected when the word ‘validation’ is invoked.

So what can we do about it?

The ML applications I have seen so far in industry exist on a spectrum between two extremes.

On one hand there are cases that aim to optimise profitability, either by reducing costs, increasing revenue, or by creating new revenue streams. and cases that require extensive documentation. In these types of cases validation often comes to mean that the ML system, as a whole, has proven to be profitable over the status quo.

On the other hand there are cases where the ML system is in place to assess or assert quality. Think product quality in manufacturing processes or transaction classification. In these types of cases, validation often comes to mean that the ML system, as a whole, has been tested and documented to be in compliance with regulations.

For the profitability centric cases, we can take advantage of DevOps tools to validate our ML systems. A/B tests and blue-green deployment are great ways to validate the ML system without disrupting operations too much. I find that the hardest part of these cases is to justify and document the business case, but once it is done, the tools are there to validate it.

For the quality centric cases, things are a bit more difficult. While there are extensive guidelines on how to implement traditional software in quality critical processes, there are little to no guidelines for machine learning systems. In a few years we might have a ‘Good Practices for Machine Learning’, but for now machine learning engineers have to jump quite a number of hoops, in order to demonstrate compliance.

Why should I care?

Even if we as data professionals are not involved in the entire architecture, it behooves us to ensure that our part integrates well in larger systems. We should also understand that just because we did cross validation and estimated model performance that it fully demonstrates the entire system; it is but the first step in a longer validation process.
